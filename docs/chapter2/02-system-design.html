System Design

Database structure, data dictionary, and the technologies, models, and algorithms used for data collection through classification in the Knowing Eye platform.

Entity-Relationship Diagram (ERD)

Knowing Eye uses a relational SQL database (SQLite in development, PostgreSQL in production). The ERD below reflects Django models under backend/features/.

JSON Schema Diagram

Omitted. Knowing Eye persists structured data in SQL tables via Django ORM. JSON fields (options, metadata, embedding) are embedded columns validated at the application layer, not standalone JSON document stores. Per capstone guidelines, JSON Schema diagrams are omitted for SQL-based databases.

The API response contract for frame analysis (used by frontend and WebSocket) follows this structure:

{
  "metrics": { "face_presence_pct", "gaze_focus_pct", "posture_compliance_pct",
               "identity_match_pct", "object_clear_pct", "overall_compliance_pct" },
  "events": [{ "event_type", "score_pct", "confidence_pct", "metadata" }],
  "alerts": [{ "type", "severity", "message", "metric_pct" }]
}

Technologies, Concepts, and Theories

The AI monitoring subsystem implements a classical computer-vision pipeline: data collection → preprocessing → feature extraction → classification/scoring. Each stage is described below with definitions and usage in Knowing Eye.

1. Data Collection

Definition: Acquisition of raw visual input from the examinee's environment during an exam session.

Technology: Browser MediaDevices.getUserMedia() API captures webcam video; canvas encodes frames as JPEG/base64 at ~1 FPS.

Transport: WebSocket (/ws/monitoring/{session_id}/) primary path; REST POST /api/monitoring/frame/ fallback.

Implementation: frontend/src/shared/hooks/use-monitoring.ts, backend/features/monitoring/consumers.py

2. Data Pre-processing

Definition: Normalization and enhancement of raw frames before model inference to improve detection stability under varying lighting and noise.

StepMethodImplementation
DecodeBase64 → BGR numpy arrayai/frame_utils.py, preprocessing/frame.py
ResizeMax width 640px (INTER_AREA)prepare_frame()
DenoisefastNlMeansDenoisingColoredConfigurable: off | light | strong
NormalizeYCbCr histogram equalizationStabilizes luminance across frames

Config: backend/ai/config/pipeline.yaml → preprocessing: section

3. Feature Extraction

Definition: Derivation of discriminative representations from preprocessed frames for downstream classification.

Feature DomainModel / LibraryExtracted Features
Face & gazeMediaPipe Face LandmarkerBounding box, 468 landmarks, head yaw/pitch angles
PostureMediaPipe Pose LandmarkerShoulder tilt ratio, spine lean ratio
ObjectsYOLOv8n (Ultralytics)Person and cell-phone detections with confidence
IdentityArcFace / InsightFace (512-D CNN embedding)Normalised face embedding vector for cosine-distance matching

Theory: CNN-based face recognition (ArcFace) learns angular-margin embeddings optimised for identity discrimination; MediaPipe provides lightweight geometric features for gaze and posture without custom training.

4. Classification & Scoring

Definition: Mapping extracted features to behavioral labels, compliance percentages, and alert severities.

StageApproachOutput
Metric normalizationThreshold-based 0–100% scoringface_presence_pct, gaze_focus_pct, posture_compliance_pct, etc.
Event classificationRule-based threshold flagsno_face, looking_away, bad_posture, object_detected, identity_mismatch, …
Temporal classificationSliding-window pattern detectionsuspicious_pattern after N flags in 60 s window
Weighted aggregationConfigurable metric_weightsoverall_compliance_pct, behavior_score (0–1)

Implementation: behavior/scoring.py, behavior/normalize.py, behavior/temporal.py

Persistence: features/behavior/services.py → BehaviorLog, Alert models → reports dashboard.
