Chapter II · Part E

System Maintenance

This section describes the post-deployment sustainment of the platform. In keeping with capstone convention, it is written entirely in the future tense.

Systems Security Plan

The system will adopt a defense-in-depth security posture across the authentication, transport, application, and data layers to protect examinee privacy, exam integrity, and institutional data throughout the operational lifecycle. Preventive, detective, and corrective controls will be maintained and reviewed on a recurring schedule.

1. Authentication & Authorization

ControlPlanned implementationMaintenance action
Password storageDjango PBKDF2 hashing will be enforcedMinimum length will be enforced; credentials will be rotated on breach
Session tokensJWT access (60 min) + refresh (7 days) with rotation & blacklist will be usedToken TTL env vars will be monitored; tokens will be revoked on compromise
RBACADMIN / EXAMINEE roles will guard all API and WS routesNew endpoints will be audited for permission classes
WebSocket authJWT in query string will be validated by JWTAuthMiddlewareWSS will be used exclusively in production

2. Transport & Network Security

  • Production traffic will be encrypted with TLS 1.2+ via Nginx; SECURE_SSL_REDIRECT=True and HSTS will be enabled when DEBUG=False.
  • CORS will be restricted to known frontend origins (CORS_ALLOWED_ORIGINS).
  • Rate limiting will be applied (DRF throttling: 60/min anonymous, 1200/min authenticated).
  • ALLOWED_HOSTS and AllowedHostsOriginValidator will prevent host-header attacks.

3. Application & Data Security

AssetThreatPlanned mitigation
Exam contentUnauthorized accessJWT + ownership checks will apply; draft exams will be hidden from examinees
Behavior logsData leakageQueries will be session-scoped; examinees will see only their own data
Face embeddingsBiometric misuseEmbeddings will be stored per session only and deleted with the session
Media uploadsMalicious filesAvatar uploads will undergo size/type validation
SecretsExposure in repo.env will remain gitignored; a strong DJANGO_SECRET_KEY will be required in production

4. Security Monitoring & Incident Response

  1. backend/logs/knowing_eye.log will be reviewed for 401/403 spikes and 500 errors.
  2. Unresolved behavior alerts will be monitored via the admin dashboard.
  3. On a suspected breach, the team will rotate DJANGO_SECRET_KEY, invalidate the JWT blacklist, and force password resets for affected accounts.
  4. Django security patches will be applied monthly via pip audit / npm audit.

Systems Maintenance Plan

Ongoing corrective, adaptive, perfective, and preventive maintenance will be carried out to preserve availability, incorporate security patches, and evolve the platform in response to user feedback and institutional policy changes.

Operational Procedures

Database Backup

  • PostgreSQL: daily pg_dump to secure storage will run
  • 30-day rolling retention will be kept; restores verified quarterly

Application Deployment

  • migrate and collectstatic will run before each release
  • Daphne workers will be restarted; frontend rebuilt (npm run build)

AI Pipeline Maintenance

  • pipeline.yaml thresholds will be updated per proctoring policy
  • python manage.py test ai.tests will be re-run after CV dependency updates

Log & Storage Hygiene

  • Rotating log (5 MB × 5 backups) will be maintained
  • Old sessions will be archived per institutional retention policy
