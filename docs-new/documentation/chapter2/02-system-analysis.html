Defines the conceptual architecture, expected system behaviors, quality attributes, and structured diagrams (use case, context DFD, Level 1 DFD) for the Knowing Eye platform.

System Architecture

Knowing Eye is a full-stack web examination platform with integrated computer-vision proctoring. The conceptual framework separates users, software components, data stores, and hardware interfaces.

Functional and Non-Functional Requirements

Requirements were derived from capstone objectives, stakeholder input from IT faculty, and iterative prototype validation. Each item is numbered for traceability to test cases (prefixes FR-, NFR-).

Functional Requirements

  1. The system shall allow users to register and authenticate using username/password with JWT access and refresh tokens.
  2. The system shall enforce role-based access control with two roles: Administrator and Examinee.
  3. Administrators shall create, update, publish, archive, and delete exams and associated questions.
  4. Administrators shall manage user accounts (activate, deactivate, assign roles).
  5. Examinees shall start an exam session for a published exam within its availability window and attempt limit.
  6. The system shall enforce one active session per examinee per exam at a time.
  7. Examinees shall answer questions within a timed session and submit responses for automatic scoring.
  8. The system shall calculate percentage score and pass/fail based on the exam passing threshold.
  9. Examinees shall enroll a reference face image before or during monitoring for identity verification.
  10. The system shall capture webcam frames and analyze behavior in near real time via WebSocket (REST fallback).
  11. The AI pipeline shall detect face absence, multiple faces, gaze deviation, poor posture, prohibited objects, and identity mismatch.
  12. The system shall persist behavior events and generate severity-classified alerts linked to the session.
  13. Administrators shall view live alerts and resolve them through the behavior API.
  14. Administrators shall access dashboard KPIs, session reports, timeseries analytics, and CSV export.
  15. Examinees shall view their own session history and results where authorized.

Non-Functional Requirements

  1. Performance: Frame analysis target ≤ 300 ms per frame at ~1–5 FPS; REST API responses ≤ 2 s under normal load.
  2. Scalability: Support multiple concurrent examinees; production uses a Redis channel layer for multi-worker WebSocket broadcast.
  3. Security: Passwords hashed (Django PBKDF2); JWT for API/WS auth; HTTPS/WSS in production; RBAC on all endpoints.
  4. Reliability: AI adapter degrades to a deterministic stub when ML dependencies are unavailable; monitoring API contract is always honored.
  5. Usability: Responsive web UI; role-specific navigation; clear exam timer and compliance feedback during monitoring.
  6. Maintainability: Feature-based modular monolith; documented REST catalog; 37 automated backend tests.
  7. Portability: Runs on Windows/Linux; SQLite for development; PostgreSQL for production without code changes.
  8. Privacy: Raw video frames are not stored by default (KE_STORE_FRAMES=False); only derived metrics and events persist.
