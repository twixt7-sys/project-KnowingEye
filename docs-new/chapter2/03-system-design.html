Chapter II · Part C

System Design

Database structure, data dictionary, and the technologies, models, and algorithms used for data collection through classification in the Knowing Eye platform.

Entity-Relationship Diagram (ERD)

Knowing Eye uses a relational SQL database (SQLite in development, PostgreSQL in production). The ERD reflects Django models under backend/features/.

JSON Schema Diagram

Omitted. Knowing Eye persists structured data in SQL tables via the Django ORM. JSON fields (options, metadata, embedding) are embedded columns validated at the application layer, not standalone JSON document stores. Per capstone guidelines, JSON Schema diagrams are omitted for SQL-based databases.

The API response contract for frame analysis (used by the frontend and WebSocket) follows this structure:

{
  "metrics": { "face_presence_pct", "gaze_focus_pct", "posture_compliance_pct",
               "identity_match_pct", "object_clear_pct", "overall_compliance_pct" },
  "events": [{ "event_type", "score_pct", "confidence_pct", "metadata" }],
  "alerts": [{ "type", "severity", "message", "metric_pct" }]
}

Technologies, Concepts, and Theories

The AI monitoring subsystem implements a classical computer-vision pipeline: data collection → preprocessing → feature extraction → classification/scoring. Each tool below states its role, version where known, and rationale.

Visual Studio Code

Primary IDE for the team, providing integrated Git, debugging, and extensions for Python, TypeScript, and React across the backend and frontend codebases.

Django 6 & DRF + Channels

Implements the backend as a feature-based modular monolith: ORM, JWT authentication, REST APIs, and the ASGI WebSocket layer (served by Daphne) that powers real-time monitoring.

React 18 + Vite

Builds the single-page application with a feature-based modular architecture: exam-taking flow, live monitoring dashboard, and reporting views with role-based protected routing.

TypeScript

Static typing across the frontend improves maintainability and catches integration errors against the API contract at compile time.

PostgreSQL (SQLite dev)

Production relational database; SQLite is used in development. The schema is portable between both engines via the Django ORM.

YOLOv8 (Ultralytics)

Detects persons and prohibited objects (e.g., mobile phones) within webcam frames, contributing object-presence signals to the behavioral compliance score.

MediaPipe

Face and pose solutions extract facial landmarks and body keypoints used for gaze estimation and posture analysis in the behavior pipeline.

ArcFace / InsightFace

512-D angular-margin embeddings provide examinee identity verification by cosine-distance matching against an enrolled reference, flagging identity mismatches.

CV Pipeline Stages

1. Data Collection

Browser getUserMedia() captures webcam video; canvas encodes JPEG/base64 at ~1 FPS. Transport: WebSocket /ws/monitoring/{session_id}/ (REST /api/monitoring/frame/ fallback).

2. Pre-processing

Base64 → BGR; resize to ≤640px; optional denoise; YCbCr histogram equalization. Config: backend/ai/config/pipeline.yaml.

3. Feature Extraction

MediaPipe (468 face landmarks, head yaw/pitch, posture ratios), YOLOv8 object detections, ArcFace identity embedding.

4. Classification & Scoring

Threshold-based 0–100% metrics, rule-based event flags, sliding-window temporal patterns, and weighted aggregation into overall_compliance_pct.
